# JavaScript for QA Automation

This repository documents my JavaScript learning path focused on QA Automation and Playwright foundations.

The goal of this repository is to practise JavaScript in a testing context instead of learning it only through abstract exercises. The examples are based on practical QA-related data such as users, login scenarios, test cases, test results, bugs, API responses, status codes and simple assertions.

## Learning Goal

The main goal is to build a solid JavaScript foundation for future automated testing with Playwright.

This repository focuses on:

* variables and data types
* conditionals
* functions
* arrays
* objects
* loops
* callbacks
* `forEach`
* `map`
* `filter`
* `find`
* JSON and API-style data
* modules
* `async/await`
* simple test-style logic

## Learning Method

This repository documents my JavaScript practice for QA Automation and Playwright foundations.

I use guided learning support, including AI-assisted explanations, hints and code review, while writing and improving the exercises myself. The goal is to understand the logic behind the code and connect JavaScript concepts with real testing scenarios.

## Why This Repository Exists

I am learning JavaScript specifically for QA Automation, not for frontend development.

Instead of focusing on frontend frameworks or unrelated projects, this repository uses QA-style examples to make JavaScript concepts easier to connect with real testing work.

Examples include:

* checking expected vs actual results
* working with test users
* filtering failed tests
* validating status codes
* reading API-like responses
* preparing reusable helper functions
* understanding logic that later appears in Playwright tests

## Repository Structure

The repository will be organized by topic and learning sessions.


## Practice Style

Each exercise is small and focused. The aim is not to build large projects immediately, but to understand how JavaScript is used in test automation.

Typical exercise format:

1. Define test data.
2. Write simple logic.
3. Check expected vs actual result.
4. Print or return a test result.
5. Refactor when needed.
